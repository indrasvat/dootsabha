v0.1.0 · Phased, Spike-First, Reality-Checked

00 Build Philosophy

Three Non-Negotiable Rules

1. Spike before you build. Every technical assumption (Codex JSONL parsing, go-plugin gRPC handshake, cobra alias behavior) gets validated with a throwaway script before it becomes load-bearing code. Phase 0 exists entirely for this. No surprises.

2. Run it, don't just test it. Unit tests are a safety net, not proof of correctness. After every phase, the agent must actually execute the binary, observe the terminal output, and verify behavior matches expectations. "It compiles" and "tests pass" are necessary but insufficient. The agent must also run the real binary against real CLIs — then screenshot / capture the output as evidence.

3. Incremental, shippable layers. Each phase produces a working binary that does something useful. Phase 1 gives you dootsabha consult. Phase 2 adds council. Phase 3 adds plugins. Each is independently valuable, not a half-finished skeleton.

Agent Discipline: The agent MUST NOT claim a phase is done without showing actual terminal output from running the binary. "I believe this works" is not evidence. $ dootsabha status followed by the actual output — that's evidence. For visual quality (colors, alignment, theme), iTerm2-driver screenshots are the proof.
go fix Discipline (Go 1.26+): Run make fix after every phase. Go 1.26's go fix (go.dev/blog/gofix) automatically rewrites code to use latest language features — new loop variable semantics, range-over-func, updated stdlib APIs. Keeps the codebase idiomatically current. Also run after every dependency update.

The Cass Memory System skill captures learnings across coding sessions. This is critical for a multi-phase project — Phase 3's go-plugin work benefits from Phase 0's spike findings, and lipgloss quirks discovered in Phase 1 shouldn't be rediscovered in Phase 4.

01 Testing Methodology

Five layers, each catching different classes of bugs. Higher layers are more valuable but slower. The agent should run layers 1-4 continuously during development, layer 5 at phase completion.

L1

Compilation & Static Analysis every save

go build ./... + go vet ./... + golangci-lint run. Catches type errors, unused imports, simple bugs. Run after every edit. Zero tolerance for warnings — fix immediately.

L2

Unit Tests every change

go test ./.... Tests individual functions in isolation. Mocked dependencies (no real CLI calls). Focus: config parsing, JSON response parsing, retry classification, subprocess arg building, flag resolution. These run in <2 seconds.

// Example: unit test for Codex JSONL parsing
func TestParseCodexJSONL(t *testing.T) {
    input := `{"type":"turn.started","turn_id":"t1"}
{"type":"item.created","item":{"type":"message","content":[{"text":"hello"}]}}
{"type":"turn.completed","turn_id":"t1"}`

    resp, err := parseCodexOutput([]byte(input))
    require.NoError(t, err)
    assert.Equal(t, "hello", resp.Content)
}
L3

Smoke Tests (Binary Execution) every phase

Build the binary, execute it, verify exit codes and output structure. No real CLIs needed — uses mock provider scripts that simulate agent behavior.

# smoke_test.sh — run after every build
set -euo pipefail

BIN="./dootsabha"

# Basic: help works
$BIN help | grep -q "दूतसभा" || fail "help missing title"
$BIN help | grep -q "council.*sabha" || fail "help missing bilingual"

# JSON mode: valid JSON on stdout, logs on stderr
output=$($BIN consult --agent mock --json "test")
echo "$output" | jq . > /dev/null || fail "invalid JSON output"

# Piped mode: no color codes
$BIN status 2>&1 | cat | grep -qP '\x1b\[' && fail "color codes in piped output"

# Exit codes
$BIN consult --agent nonexistent "test"; [ $? -ne 0 ] || fail "bad exit code"

echo "✓ All smoke tests passed"
L4

Integration Tests (Real CLIs) every phase

Run against actual claude, codex, gemini CLIs. Validates real subprocess behavior, JSON parsing, auth detection, timeout handling. Requires CLIs installed and authenticated. Costs actual API tokens — keep prompts tiny.

# integration_test.sh — run with real CLIs
set -euo pipefail

BIN="./dootsabha"
PROMPT="Reply with exactly: PONG"

echo "── Claude ──"
result=$($BIN consult --agent claude --json "$PROMPT")
echo "$result" | jq -r '.content' | grep -qi "PONG" || fail "claude: no PONG"
echo "$result" | jq -e '.duration_ms > 0' > /dev/null || fail "claude: no duration"
echo "$result" | jq -e '.model' > /dev/null || fail "claude: no model"

echo "── Codex ──"
result=$($BIN consult --agent codex --json "$PROMPT")
echo "$result" | jq -r '.content' | grep -qi "PONG" || fail "codex: no PONG"

echo "── Gemini ──"
result=$($BIN consult --agent gemini --json "$PROMPT")
echo "$result" | jq -r '.content' | grep -qi "PONG" || fail "gemini: no PONG"

echo "── Status ──"
$BIN status --json | jq -e '.providers | length == 3' > /dev/null

echo "✓ All integration tests passed"
L5

Acceptance Tests (End-to-End + Visual) phase completion

Full user-facing scenarios that exercise the entire pipeline, plus iTerm2-driver visual verification. The agent must run these, take screenshots via iterm2-driver, and visually inspect rendering quality — colors, alignment, Unicode, degradation.

02 Terminal UX Standards

This is not optional polish — it's a core quality bar for the MVP. Every CLI output must be beautiful in a terminal and correct when piped.

Good Unix Citizenship

PrincipleImplementationValidation
stdout = data, stderr = logs All structured output (JSON, results) goes to stdout. All status messages, spinners, progress, warnings go to stderr. dootsabha consult --json "test" | jq . must produce valid JSON
Meaningful exit codes 0=success, 1=general error, 2=usage error (bad flags), 3=provider error (CLI failed), 4=timeout, 5=partial result (some agents failed in council) Test each exit code path explicitly
NO_COLOR + pipe detection Respect NO_COLOR env var. Auto-detect if stdout is a TTY via os.Stdout.Fd() + isatty. No color/spinner/progress when piped. dootsabha status | cat | grep -P '\x1b\[' must find nothing
Stderr for human feedback Spinners, progress bars, "Invoking claude..." messages — all to stderr. Never pollute stdout with status noise. dootsabha consult "test" > /dev/null should show progress on terminal
Ctrl+C graceful shutdown Catch SIGINT/SIGTERM. Kill child processes (process groups). Print summary of what completed. Clean non-zero exit. Hit Ctrl+C mid-council: see "Interrupted. Claude completed, codex cancelled." Not a Go stack trace.
Width-aware formatting Detect terminal width via term.GetSize(). Tables and wrapped text respect it. Degrade to simple output at <60 cols. Resize terminal to 40 cols, run dootsabha status — output must not wrap hideously

P0 Phase 0 — Spikes & Assumption Validation

BEFORE ANY PRODUCTION CODE · ~1.5 days · throwaway code only

Every spike is a standalone, disposable script/program that proves a single assumption. If a spike fails, we redesign before writing production code. Spikes are committed to _spikes/ for reference, never imported.

Spike 0.1 — Codex CLI JSONL Parsing

Risk: Codex outputs JSONL events, not a single JSON object. We need to reliably extract the final response content from the event stream. This is the highest-risk parsing challenge.

Throwaway script:

// _spikes/codex_jsonl/main.go
// Run: codex exec "Reply with exactly: PONG" --json --ephemeral --skip-git-repo-check 2>stderr.log | go run main.go
// Prove: prints parsed content, model, all event types seen
  • Capture raw JSONL from codex exec --json to a file
  • Document every event type observed (turn.started, item.created, turn.completed, etc.)
  • Identify which event(s) contain the final text response
  • Handle multi-line content, embedded newlines, Unicode
  • Handle error events gracefully
  • Write a parseCodexOutput([]byte) (*Response, error) that works on the captured data

Done when: go run main.go < captured_output.jsonl prints the correct content.

Spike 0.2 — Claude CLI JSON Output

Risk: Lower than Codex (single JSON object), but need to verify exact field names, nested structure, and edge cases (errors, empty responses).

  • Run claude -p "PONG" --output-format json --dangerously-skip-permissions --no-session-persistence
  • Capture and document the full JSON schema
  • Test with an error case (invalid model name) — what does the JSON look like?
  • Test cost_usd field — is it always present? Is it accurate?
  • Write parseClaudeOutput([]byte) (*Response, error)

Done when: Parser works on both success and error JSON.

Spike 0.3 — Gemini CLI JSON Output

Risk: The --yolo bug (#13561), positional prompt syntax, and JSON schema need validation.

  • Run gemini "PONG" --output-format json --model gemini-3-pro --approval-mode yolo
  • Confirm --approval-mode yolo works (not --yolo)
  • Capture and document JSON schema (response, stats fields)
  • Test error case (invalid model)
  • Write parseGeminiOutput([]byte) (*Response, error)

Done when: Parser works. --approval-mode yolo confirmed working.

Spike 0.4 — hashicorp/go-plugin gRPC Handshake

Risk: First-time go-plugin setup is fiddly. Proto compilation, handshake cookies, binary discovery, version negotiation — lots of moving parts.

  • Create a minimal host + plugin pair with a single gRPC method (Ping() → "pong")
  • Proto file → protoc → generated Go code (document exact protoc invocation)
  • Plugin binary builds and runs standalone
  • Host discovers and loads plugin binary
  • gRPC call succeeds across the process boundary
  • Plugin crash → host recovers gracefully (no panic)
  • Measure overhead: how long does handshake + single call take? (<100ms acceptable)

Done when: Host prints "pong" from plugin. Plugin crash test passes.

Spike 0.5 — Subprocess Management Patterns

Risk: Context cancellation, process group cleanup, stdout/stderr splitting under concurrent execution.

  • Launch 3 concurrent subprocesses with errgroup
  • Each sleeps for different durations
  • Cancel one via context timeout — verify it dies and others continue
  • Kill with SIGINT — verify all children get cleaned up (no orphans)
  • Verify stdout/stderr separation under concurrent writes
  • Verify Setpgid: true + process group kill works on macOS

Done when: ps aux | grep spike shows zero orphans after Ctrl+C.

Spike 0.6 — Cobra Alias Behavior

Risk: Low, but need to confirm that cobra's Aliases field works the way we need for bilingual subcommands and that help output formats correctly.

  • Create a minimal cobra app with council command and Aliases: []string{"sabha"}
  • Verify app sabha works identically to app council
  • Verify help output shows both names
  • Verify flag aliases (--agents / --dootas) via pflag
  • Test unknown-command fallback for extension discovery

Done when: Both names work. Help shows them. Unknown command triggers fallback handler.

Spike 0.7 — Terminal UX Foundations

Risk: lipgloss behavior under pipe, NO_COLOR, narrow terminals.

  • Build a tiny program with lipgloss styled table + spinner
  • Run in terminal — colors and spinner work
  • Pipe to cat — no ANSI codes, no spinner artifacts
  • Set NO_COLOR=1 — styled but no colors
  • Set terminal to 40 columns — output doesn't break
  • Verify lipgloss.HasDarkBackground() detection

Done when: All 4 contexts produce clean, readable output.

## Project Conventions
- **Go 1.26+** — use latest language features. Run `make fix` after changes.
- **Always use `make` targets** — never raw go/golangci-lint/protoc commands.
  If you discover a new useful command, add it to the Makefile first, then use it.
- **Pre-commit**: `make check` (= fmt + fix + lint + vet + test + smoke)
- **Test pyramid**: L1 (compile) → L2 (unit) → L3 (smoke) → L4 (integration) → L5 (acceptance+visual)
- **Prove it works**: paste actual terminal output. For visual quality, use iTerm2 screenshots.
- **Episodic memory**: after significant discoveries/gotchas, save to `cm`.

## Code Style
- Go 1.26+ idioms: range-over-func, enhanced loop vars, new stdlib APIs
- Run `go fix` via `make fix` regularly — especially after dep updates
- golangci-lint strict mode — zero warnings policy
- slog for all logging, stderr only
- lipgloss for all terminal rendering
- Errors: wrap with context, classify transient vs permanent
- Tests: testify for assertions, table-driven where possible

P5.3 — Claude Code SKILL for दूतसभा

Since दूतसभा is a tool used by coding agents, it needs a first-class Claude Code SKILL so that any agent can discover and use it effectively. This lives in skill/ at the project root.

  • skill/SKILL.md — progressive disclosure, optimized for Claude Code context window
  • skill/metadata.json — trigger words, categories, version
  • skill/examples/ — copy-pasteable usage patterns

SKILL.md — Design Principles

Progressive disclosure: The SKILL must be layered so that an agent that reads only the first 20 lines can use dootsabha consult correctly. Deeper reading unlocks council, review, extensions, plugins, advanced flags — each layer additive, never invalidating previous layers.

Trigger-optimized: Metadata includes trigger words that fire when agents encounter scenarios where दूतसभा helps — multi-agent comparison, code review, consensus, benchmarking.

Zero-ambiguity flag docs: Every flag includes its exact syntax, default, Sanskrit alias, and a concrete example. No agent should ever have to guess.

# दूतसभा — The Council of Agents

## Quick Start (read this first)

```bash
# Check which agents are available
dootsabha status

# Ask a single agent
dootsabha consult "explain this error: connection refused on port 5432"

# Full council: all agents discuss, peer-review, synthesize
dootsabha council "refactor the auth middleware for better testability"

# One agent reviews another's work
dootsabha review "optimize the database queries in models/user.go"
```

## Core Subcommands

| Command | Alias | What it does |
|---------|-------|-------------|
| `council` | `sabha` | All agents → peer review → synthesis |
| `consult` | `paraamarsh` | Single agent query |
| `review` | `sameeksha` | One agent reviews another |
| `status` | `sthiti` | Health check all agents |

## Machine-Consumable Output

All commands support `--json`. Stdout is pure JSON data, stderr is logs.

```bash
# Pipe council output to jq
result=$(dootsabha council --json "fix the failing tests")
echo "$result" | jq -r '.synthesis.content'

# Check cost
echo "$result" | jq '.meta.total_cost_usd'
```

## Key Flags

| Flag | Alias | Default | Purpose |
|------|-------|---------|---------|
| `--agent` | `--doota` | claude | Provider for consult |
| `--agents` | `--dootas` | claude,codex,gemini | Providers for council |
| `--chair` | `--adhyaksha` | claude | Synthesis agent |
| `--json` | — | false | JSON output mode |
| `--timeout` | `--kaalseema` | 5m | Per-agent timeout |

## When to Use दूतसभा

- **Want multiple perspectives** on a design decision → `council`
- **Want a code review** from a different AI → `review`
- **Want the best single answer** → `consult` (just picks one agent)
- **Building a CI pipeline** that needs structured output → `--json`
- **Benchmarking agents** on your specific codebase → `dootsabha bench` (extension)

## Advanced: Extensions

दूतसभा is extensible. Any binary named `dootsabha-{name}` on $PATH 
becomes `dootsabha {name}`. Extensions receive rich context via env 
vars and a JSON context file. See `examples/extension-dev.md`.
