01 Architecture Overview

दूतसभा follows a small core + plugin shell architecture. The core handles cross-cutting concerns (config, logging, metrics, subprocess lifecycle, plugin discovery) while every feature — providers, output modes, synthesis strategies — is a plugin implementing a well-defined Go interface. Plugins are standalone binaries communicating over gRPC via hashicorp/go-plugin.

Core Responsibilities

Config Manager

Unified config via viper: YAML file → env vars → CLI flags (highest precedence). Supports per-provider model overrides, YOLO defaults, and plugin paths.

Plugin Manager

Discovers, loads, and manages plugin binaries via hashicorp/go-plugin. Handles handshake protocol, health checks, version negotiation, and graceful shutdown.

Session Manager

Owns the state machine for each request lifecycle. Coordinates dispatch across providers, collects responses, triggers peer review, and orchestrates synthesis.

Subprocess Runner

Wraps os/exec with context-aware cancellation, structured log capture (stdout/stderr splitting), timeout enforcement, and process cleanup on SIGINT/SIGTERM.

Design Principles

SOLID + OCP: The core is closed for modification, open for extension. Every new capability (provider, strategy, formatter, subcommand, lifecycle hook) is a plugin — zero core changes needed.

Dual-Track Extensibility: gRPC plugins (Provider, Strategy, Formatter, Hook) for structured, typed contracts with the core. Extension plugins (dootsabha-*) for adding entirely new subcommands — any language, zero ceremony, gh/kubectl pattern.

Structs + Interfaces: Concrete types implement thin interfaces. No interface pollution. Plugin interfaces are the contract, structs are the implementation.

Fail-Fast + Recover: Plugin crashes don't take down the core (process isolation via go-plugin). Transient failures get retries with exponential backoff; permanent failures fail immediately with actionable error messages.

Agent-First DX: Every output supports --json for machine consumption. Stdout carries data, stderr carries logs. Exit codes are meaningful. Streaming via --watch.

02 Plugin System

Built on hashicorp/go-plugin (battle-tested by Terraform, Vault, Nomad, Packer — 3,570+ Go projects). Each plugin is a standalone binary communicating with the core over gRPC. Plugins live in ~/.dootsabha/plugins/ or a configured path.

Plugin Interface (Go)

// ─── Provider Plugin ─── The primary interface for coding agents.
type Provider interface {
    // Name returns the friendly name (e.g., "claude", "codex", "gemini").
    Name() string

    // Capabilities returns supported features (streaming, JSON output, etc.).
    Capabilities() ProviderCaps

    // Invoke sends a prompt and returns structured output.
    // ctx carries deadline, cancellation, and trace ID.
    Invoke(ctx context.Context, req *InvokeRequest) (*InvokeResponse, error)

    // HealthCheck verifies the underlying CLI is installed and authenticated.
    HealthCheck(ctx context.Context) (*HealthStatus, error)
}

// ─── Strategy Plugin ─── Orchestration modes (council, consult, etc.).
type Strategy interface {
    Name() string
    Execute(ctx context.Context, req *StrategyRequest) (*StrategyResponse, error)
}

// ─── Formatter Plugin ─── Output rendering (JSON, table, markdown, diff).
type Formatter interface {
    Name() string
    Format(ctx context.Context, data *FormatInput) ([]byte, error)
}

// ─── Hook Plugin ─── Lifecycle middleware (pre/post invoke, synthesis, etc.).
// Hooks run in registration order. Returning an error from a "Pre" hook
// aborts the operation. "Post" hook errors are logged but don't abort.
type Hook interface {
    Name() string

    // Which lifecycle events this hook subscribes to.
    Events() []HookEvent

    // PreInvoke runs before each agent invocation.
    // Can modify the request (prompt rewriting, injection, etc.).
    PreInvoke(ctx context.Context, req *InvokeRequest) (*InvokeRequest, error)

    // PostInvoke runs after each agent invocation.
    // Can inspect/transform the response (redaction, enrichment, etc.).
    PostInvoke(ctx context.Context, req *InvokeRequest, resp *InvokeResponse) (*InvokeResponse, error)

    // PreSynthesis runs before the chair synthesizes final output.
    PreSynthesis(ctx context.Context, responses []*InvokeResponse) ([]*InvokeResponse, error)

    // PostSession runs after the full session completes (metrics, cleanup, etc.).
    PostSession(ctx context.Context, session *SessionSummary) error
}

type HookEvent string
const (
    HookPreInvoke     HookEvent = "pre_invoke"
    HookPostInvoke    HookEvent = "post_invoke"
    HookPreSynthesis  HookEvent = "pre_synthesis"
    HookPostSession   HookEvent = "post_session"
)

// ─── Extension Plugin ─── NOT a gRPC plugin.
// Extensions are standalone binaries on $PATH named "dootsabha-{name}".
// They register as "dootsabha {name}" subcommands (gh-style).
// No interface to implement — just a binary that accepts args.
//
// Context is delivered via 3 tiers:
//
// Tier 1 (env vars): DOOTSABHA_BIN, DOOTSABHA_CONFIG, DOOTSABHA_SESSION_ID,
//   DOOTSABHA_VERSION, DOOTSABHA_DATA_DIR, DOOTSABHA_CACHE_DIR,
//   DOOTSABHA_STATE_DIR, DOOTSABHA_PLUGINS_DIR, DOOTSABHA_LOG_LEVEL,
//   DOOTSABHA_LOG_FORMAT, DOOTSABHA_OUTPUT_FORMAT, DOOTSABHA_TTY,
//   DOOTSABHA_COLOR, DOOTSABHA_TERM_WIDTH, DOOTSABHA_WORK_DIR,
//   DOOTSABHA_GIT_ROOT, DOOTSABHA_GIT_BRANCH, DOOTSABHA_CONTEXT_FILE
//
// Tier 2 (context file): JSON at $DOOTSABHA_CONTEXT_FILE with full resolved
//   config, provider registry (health + capabilities), workspace/git state,
//   directory paths, terminal info, installed extensions/plugins list.
//   Core writes this before exec, cleans up after exit.
//
// Tier 3 (callback): Extensions can invoke $DOOTSABHA_BIN directly to
//   run agents, council sessions, status checks, etc. Session ID propagates
//   automatically for correlated metrics/traces.
//
// Extensions inherit stdin/stdout/stderr directly (no capture).
// Exit code is forwarded to the caller.

// ─── Common Types ───
type InvokeRequest struct {
    Prompt       string            `json:"prompt"`
    Model        string            `json:"model,omitempty"`
    MaxTurns     int               `json:"max_turns,omitempty"`
    Temperature  float64           `json:"temperature,omitempty"`
    OutputFormat string            `json:"output_format"` // "json" | "text" | "stream-json"
    ExtraArgs    []string          `json:"extra_args,omitempty"`
    Env          map[string]string `json:"env,omitempty"`
    WorkDir      string            `json:"work_dir,omitempty"`
    Timeout      time.Duration     `json:"timeout,omitempty"`
}

type InvokeResponse struct {
    Content    string             `json:"content"`
    RawJSON    json.RawMessage    `json:"raw_json,omitempty"`
    Provider   string             `json:"provider"`
    Model      string             `json:"model"`
    SessionID  string             `json:"session_id,omitempty"`
    CostUSD    float64            `json:"cost_usd,omitempty"`
    TokensIn   int                `json:"tokens_in,omitempty"`
    TokensOut  int                `json:"tokens_out,omitempty"`
    DurationMs int64              `json:"duration_ms"`
    ExitCode   int                `json:"exit_code"`
    Stderr     string             `json:"stderr,omitempty"`
}

type ProviderCaps struct {
    SupportsJSON      bool     `json:"supports_json"`
    SupportsStreaming  bool     `json:"supports_streaming"`
    SupportedModels   []string `json:"supported_models"`
    DefaultModel      string   `json:"default_model"`
    MaxContextTokens  int      `json:"max_context_tokens"`
}

type HealthStatus struct {
    OK        bool   `json:"ok"`
    Version   string `json:"version,omitempty"`
    CLIPath   string `json:"cli_path,omitempty"`
    AuthValid bool   `json:"auth_valid"`
    Error     string `json:"error,omitempty"`
}

type SessionSummary struct {
    SessionID   string            `json:"session_id"`
    Strategy    string            `json:"strategy"`
    Providers   []string          `json:"providers"`
    TotalCost   float64           `json:"total_cost_usd"`
    TotalTokens int               `json:"total_tokens"`
    DurationMs  int64             `json:"duration_ms"`
    Errors      []string          `json:"errors,omitempty"`
}

Plugin Lifecycle

Process Isolation: Each plugin runs as a separate OS process. A panicking plugin cannot crash the core. Plugins are discovered at startup and can be upgraded independently by replacing the binary and restarting. While hashicorp/go-plugin doesn't support hot-reload, plugin processes are lightweight and restart in <100ms.

Extension Plugins — The gh/kubectl Pattern

How it works: Any executable on $PATH or in ~/.dootsabha/plugins/ named dootsabha-{name} automatically becomes available as dootsabha {name}. No gRPC, no interface, no Go required — just a binary that accepts args.

Precedent: This is the exact pattern used by gh (GitHub CLI extensions: gh-dash, gh-copilot), kubectl (krew: kubectl-ctx, kubectl-ns), and git (git-lfs, git-flow). Battle-tested, zero-friction, language-agnostic.

Tier 2 — Context File (JSON) rich structured data

Before exec'ing an extension, the core writes a temporary JSON file containing the full runtime context. The path is in DOOTSABHA_CONTEXT_FILE. This solves the fundamental problem with env vars — complex nested data, lists, and provider registries don't survive shell escaping.

Why a file, not stdin? Extensions need stdin for interactive input (TUIs, prompts). A temp file is language-agnostic, parseable by jq in Bash or json.load() in Python, and has no size limits. Core cleans it up after the extension exits.

Context File in Practice — Bash & Python

# Bash: read provider health with jq
#!/usr/bin/env bash
CTX=$(cat "$DOOTSABHA_CONTEXT_FILE")
HEALTHY=$(echo "$CTX" | jq -r '[.providers | to_entries[] | select(.value.healthy)] | map(.key) | join(",")')
echo "Healthy agents: $HEALTHY"
MODEL=$(echo "$CTX" | jq -r '.config.providers.claude.model')
echo "Claude model: $MODEL"

# Python: full context access
#!/usr/bin/env python3
import json, os
with open(os.environ["DOOTSABHA_CONTEXT_FILE"]) as f:
    ctx = json.load(f)
for name, info in ctx["providers"].items():
    status = "✓" if info["healthy"] else f"✗ {info.get('error','')}"
    print(f"  {name}: {status} ({info['capabilities']['default_model']})")

Tier 3 — Core Callback full power

Extensions can invoke $DOOTSABHA_BIN directly for operations that need the core's orchestration — invoking agents, running council sessions, checking live status. This is the gh api equivalent.

Design choice: Callback via subprocess (not a local socket/API). Keeps the contract dead simple — extensions just shell out to the same binary the user installed. No daemon, no port conflicts, no auth tokens. The session ID propagates automatically via DOOTSABHA_SESSION_ID so nested calls are correlated in metrics/traces. 
#!/usr/bin/env bash
# dootsabha-bench: benchmark extension using Tier 3 callbacks
DS="$DOOTSABHA_BIN"
PROMPT="$1"
ROUNDS="${2:-3}"

echo "Benchmarking '$PROMPT' across all healthy providers ($ROUNDS rounds)..."

# Read healthy providers from context file (Tier 2)
PROVIDERS=$(jq -r '[.providers | to_entries[] | select(.value.healthy) | .key] | .[]' "$DOOTSABHA_CONTEXT_FILE")

for provider in $PROVIDERS; do
  echo "── $provider ──"
  for i in $(seq 1 "$ROUNDS"); do
    # Tier 3: callback into core to invoke agents
    result=$("$DS" consult --agent "$provider" --json "$PROMPT")
    cost=$(echo "$result" | jq -r '.cost_usd // 0')
    ms=$(echo "$result" | jq -r '.duration_ms')
    tokens=$(echo "$result" | jq -r '.tokens_out // 0')
    echo "  Round $i: ${ms}ms, ${tokens} tokens, \$${cost}"
  done
done

Hook Plugins — Lifecycle Middleware

How it works: Hook plugins subscribe to lifecycle events (pre_invoke, post_invoke, pre_synthesis, post_session) and run as middleware in registration order. They can transform requests/responses or perform side effects.

Use cases: Prompt injection/templating, PII redaction, cost alerts, Slack notifications on completion, audit logging, response caching, A/B testing different models.

YOLO Defaults — Rationale

All three providers default to their maximum-permission modes. This eliminates interactive approval prompts, which is essential for non-interactive subprocess orchestration. The alternative (interactive approvals) would cause the subprocess to hang waiting for stdin.

Override per-provider via config:

  claude:
    yolo: false                          # Reverts to interactive approvals
    allowed_tools: ["Bash(git *)"]       # Restrict tool access
  codex:
    sandbox: "workspace-write"           # Downgrade from full access
  gemini:
    approval_mode: "auto"                # Downgrade from yolo

Error Classification

Transient → Auto-Retry

Rate limiting (429), network timeouts, temporary auth failures, process killed by OOM. Up to 2 retries with exponential backoff (1s, 4s). Logged at WARN.

Permanent → Fail Fast

10 Plugin Development Guide

Writing a New Provider Plugin

Every provider plugin is a standalone Go binary that implements the Provider gRPC service. Here's the minimal skeleton:

// plugins/my-agent/main.go
package main

import (
    "github.com/hashicorp/go-plugin"
    shared "github.com/indrasvat/dootsabha/proto"
)

type MyAgentProvider struct{}

func (p *MyAgentProvider) Name() string { return "my-agent" }

func (p *MyAgentProvider) Capabilities() *shared.ProviderCaps {
    return &shared.ProviderCaps{
        SupportsJson:     true,
        SupportsStreaming: false,
        SupportedModels:  []string{"model-v1", "model-v2"},
        DefaultModel:     "model-v1",
        MaxContextTokens: 128000,
    }
}

func (p *MyAgentProvider) Invoke(ctx context.Context, req *shared.InvokeRequest) (*shared.InvokeResponse, error) {
    // 1. Build CLI command from req
    // 2. Execute via os/exec.CommandContext
    // 3. Parse output (JSON/text)
    // 4. Return InvokeResponse
}

func (p *MyAgentProvider) HealthCheck(ctx context.Context) (*shared.HealthStatus, error) {
    // Check CLI is installed, authenticated, reachable
}

func main() {
    plugin.Serve(&plugin.ServeConfig{
        HandshakeConfig: shared.Handshake,
        Plugins: map[string]plugin.Plugin{
            "provider": &shared.ProviderGRPCPlugin{Impl: &MyAgentProvider{}},
        },
        GRPCServer: plugin.DefaultGRPCServer,
    })
}

Writing an Extension Plugin (Any Language)

Extensions are the simplest plugin type — just a binary named dootsabha-{name}. No Go required, no gRPC, no interface. Write it in Python, Rust, Bash, whatever.

#!/usr/bin/env bash
# dootsabha-hello — simplest possible extension
# Install: chmod +x dootsabha-hello && mv to ~/.dootsabha/plugins/ or PATH

# Tier 1: env vars (always available)
echo "Hello from दूतसभा extension!"
echo "Core:    ${DOOTSABHA_VERSION} (${DOOTSABHA_BIN})"
echo "Session: ${DOOTSABHA_SESSION_ID}"
echo "Args:    $@"

# Tier 2: context file (rich structured data)
if [ -n "$DOOTSABHA_CONTEXT_FILE" ]; then
  AGENTS=$(jq -r '[.providers | to_entries[] | select(.value.healthy) | .key] | join(", ")' "$DOOTSABHA_CONTEXT_FILE")
  echo "Healthy: $AGENTS"
fi

# Tier 3: callback (invoke agents via core)
if [ -n "$1" ]; then
  echo "Asking claude: $1"
  "$DOOTSABHA_BIN" consult --agent claude --json "$1" | jq -r '.content'
fi

# Usage — automatically discovered as a subcommand:
$ dootsabha hello "what is a goroutine?"
Hello from दूतसभा extension!
Core:    0.1.0 (/usr/local/bin/dootsabha)
Session: ds_a1b2c3d4
Args:    what is a goroutine?
Healthy: claude, codex
Asking claude: what is a goroutine?
A goroutine is a lightweight thread of execution managed by the Go runtime...
Key difference from gRPC plugins: Extensions inherit the terminal directly — they get stdin/stdout/stderr unmediated. This means they can run interactive TUIs, prompt for input, display spinners, etc. gRPC plugins can't do this because the core captures their output.

Writing a Hook Plugin

Hook plugins implement the Hook gRPC interface and subscribe to specific lifecycle events.

// plugins/cost-guard/main.go
package main

import (
    "context"
    "fmt"
    "github.com/hashicorp/go-plugin"
    shared "github.com/indrasvat/dootsabha/proto"
)

type CostGuardHook struct {
    budgetUSD float64
    spentUSD  float64
}

func (h *CostGuardHook) Name() string { return "cost-guard" }

func (h *CostGuardHook) Events() []shared.HookEvent {
    return []shared.HookEvent{shared.HookPreInvoke, shared.HookPostInvoke}
}

func (h *CostGuardHook) PreInvoke(ctx context.Context, req *shared.InvokeRequest) (*shared.InvokeRequest, error) {
    if h.spentUSD >= h.budgetUSD {
        return nil, fmt.Errorf("budget exhausted: spent $%.2f of $%.2f", h.spentUSD, h.budgetUSD)
    }
    return req, nil // pass through unmodified
}

func (h *CostGuardHook) PostInvoke(ctx context.Context, req *shared.InvokeRequest, resp *shared.InvokeResponse) (*shared.InvokeResponse, error) {
    h.spentUSD += resp.CostUsd
    return resp, nil
}

// ... (PreSynthesis, PostSession return unchanged / nil)

func main() {
    plugin.Serve(&plugin.ServeConfig{
        HandshakeConfig: shared.Handshake,
        Plugins: map[string]plugin.Plugin{
            "hook": &shared.HookGRPCPlugin{Impl: &CostGuardHook{budgetUSD: 5.0}},
        },
        GRPCServer: plugin.DefaultGRPCServer,
    })
}

Subprocess Best Practices (Go)

func (p *ClaudeProvider) execCLI(ctx context.Context, args []string) (*ExecResult, error) {
    cmd := exec.CommandContext(ctx, "claude", args...)

    // Separate stdout (data) from stderr (logs)
    var stdout, stderr bytes.Buffer
    cmd.Stdout = &stdout
    cmd.Stderr = &stderr

    // Set process group for clean cleanup
    cmd.SysProcAttr = &syscall.SysProcAttr{Setpgid: true}

    // Working directory
    cmd.Dir = p.workDir

    // Environment (inherit + extras)
    cmd.Env = append(os.Environ(), p.extraEnv...)

    start := time.Now()
    err := cmd.Run()
    elapsed := time.Since(start)

    result := &ExecResult{
        Stdout:     stdout.Bytes(),
        Stderr:     stderr.String(),
        ExitCode:   cmd.ProcessState.ExitCode(),
        DurationMs: elapsed.Milliseconds(),
    }

    if err != nil {
        if ctx.Err() == context.DeadlineExceeded {
            return result, fmt.Errorf("timeout after %v: %w", p.timeout, err)
        }
        if ctx.Err() == context.Canceled {
            return result, fmt.Errorf("canceled: %w", err)
        }
        return result, fmt.Errorf("exit code %d: %s", result.ExitCode, result.Stderr)
    }

    return result, nil
}

11 Usage Examples

Quick Start

# Check all agents are healthy
dootsabha status

# Ask claude a question (default agent, default model)
dootsabha consult "explain this Kubernetes CrashLoopBackOff"

# Same in Sanskrit
dootsabha paraamarsh "explain this Kubernetes CrashLoopBackOff"

# Full council deliberation
dootsabha council "refactor auth middleware for better testability"

# Council with streaming
dootsabha sabha --watch "design a rate limiter for our API gateway"

# Review: codex writes, claude reviews
dootsabha review "optimize the database queries in pkg/store"

# Use a specific model
dootsabha consult --agent claude --model opus-4-6 "complex architecture decision"

# JSON output for agent consumption
dootsabha council --json "implement retry logic" | jq .

# Override council members
dootsabha council --agents claude,codex "only need two opinions"

# Change the synthesis chair
dootsabha sabha --adhyaksha gemini "want gemini to synthesize"

Agent / CI Usage

# Machine-readable output (stdout=JSON, stderr=logs)
result=$(dootsabha council --json "fix failing tests" 2>/tmp/ds.log)
echo "$result" | jq -r '.synthesis.content'

# With timeout
dootsabha consult --timeout 2m --json "quick question"

# Pipe prompt from file
cat prompt.md | dootsabha council --json

# Non-zero exit codes for failures
dootsabha consult "query" || echo "Failed with exit code $?"
